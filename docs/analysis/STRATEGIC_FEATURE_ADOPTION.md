## Part I: Philosophy

### The Problem with Most CRMs

Most CRMs are built by engineers who think in **databases**: Tours, Customers, Bookings, Schedules, Guides. They create a module for each table and call it done.

But tour operators don't think in databases. They think in **moments**:

- *"A customer is on the phone right now. I need to book them in 60 seconds."*
- *"It's 6 AM. What do I need to know before today's tours?"*
- *"Someone just canceled. Who's on the waitlist?"*
- *"Is Mike available to cover Sarah's tour tomorrow?"*

**The gap between how software is built and how humans work is why most CRMs feel like punishment.**

Our CRM will be different. We will design for **moments**, not modules.

---

### Design Principles

These are non-negotiable. Every decision passes through these filters.

#### 1. Three Clicks to Anything

Steve Jobs demanded that any function on the Mac be reachable in three intuitive clicks. If designers couldn't achieve this, the feature was redesigned or cut.

**Our Rule:** Any core task completes in 3 interactions or fewer.

| Task | Interactions |
|------|-------------|
| Create a booking | Search customer â†’ Select tour/date â†’ Confirm |
| Find a customer | Type name in search â†’ Click result |
| Check today's schedule | Open app (it's the first thing you see) |
| Assign a guide | Click schedule â†’ Select guide from dropdown |

#### 2. One Primary Action Per Screen

Multiple equal-weight buttons create decision paralysis. Every screen has exactly one obvious next step.

| Screen | Primary Action | Secondary Actions |
|--------|---------------|-------------------|
| Dashboard | None (information screen) | Quick actions in cards |
| Booking Detail | Send Confirmation | Edit, Cancel (smaller) |
| Customer Profile | New Booking | Edit, Message (smaller) |
| Empty Schedule | Add Tour Date | (only action visible) |

#### 3. Recognition Over Recall

Don't make users remember. Show them options.

- **Bad:** "Enter customer ID: ____"
- **Good:** Search box that shows recent customers + suggestions as you type

- **Bad:** "Enter tour code: ____"
- **Good:** Visual tour cards with availability shown

#### 4. Invisible Complexity

Power users need advanced features. New users need simplicity. Both should be served by the same interface.

- Default view: Essential fields only
- "More options" expands advanced settings
- Keyboard shortcuts for power users, never required

#### 5. The Interface Should Disappear

When using a great tool, you don't think about the toolâ€”you think about your work. Our CRM should feel like an extension of the operator's mind, not a system to be operated.

**Key Design Decisions:**
- **Starts with customer**, not tour (matches phone call flow: "Hi, I'd like to book...")
- **Recent customers shown immediately** (80% of bookings are repeat customers)
- **Tours shown as cards with availability** (not a dropdown requiring recall)
- **Inline +/- for guests** (no typing, no tab-tab-tab)
- **Advanced options hidden by default** (special requests, promo codes)
- **Single "Complete Booking" button** (one primary action)

**Time:** 3 clicks, ~15 seconds for repeat customer.

- Must click into each schedule to check guide status
- Must click into each booking to check payment status
- Problems discovered when it's too late

**Solution: The Attention-First Dashboard**

The dashboard is not a report. It's a **command center** that surfaces what matters.

## Part VI: Visual Language

### Color System

Color conveys meaning, not decoration.

| Color | Meaning | Usage |
|-------|---------|-------|
| **Blue** | Primary action, links | Buttons, clickable text |
| **Green** | Success, confirmed, good | Status badges, success messages |
| **Yellow** | Warning, pending, attention | Status badges, alerts |
| **Red** | Error, danger, urgent | Errors, destructive actions |
| **Gray** | Neutral, disabled, secondary | Borders, secondary text, disabled |

### Typography Hierarchy

```
Page Title        24px  Bold      (One per page)
Section Title     18px  Semibold  (Card headers)
Label             14px  Medium    (Form labels, table headers)
Body              14px  Regular   (Default text)
Caption           12px  Regular   (Helper text, timestamps)
```

### Spacing System

Use 4px base unit. All spacing is multiples of 4.

```
Micro:   4px   (inline elements)
Small:   8px   (related elements)
Medium:  16px  (sections within a card)
Large:   24px  (between cards)
XLarge:  32px  (page sections)
```

### Task Completion Benchmarks

| Task | Target Time | Max Clicks |
|------|-------------|------------|
| Create booking (existing customer) | < 30 seconds | 3 |
| Create booking (new customer) | < 60 seconds | 5 |
| Find a customer by name | < 5 seconds | 2 |

| Send booking confirmation | < 5 seconds | 2 |

### Error Prevention

| Error Type | Target Rate |
|------------|-------------|
| Form validation errors | < 5% of submissions |
| User asks "how do I...?" | < 1 per 100 active users/month |
| Abandoned booking flow | < 10% |

### The Steve Jobs Test

For every feature, ask:
1. Can it be reached in 3 clicks?
2. Would a first-time user understand it in 2 seconds?
3. Does it make the product simpler or more complex?
4. Could you explain it to someone who's never used software?

If any answer is no, redesign.

## Part IX: What We DON'T Build

Saying no is as important as saying yes.

### Features We Won't Add

| Feature | Why Not |
|---------|---------|
| Customizable dashboards | Complexity explosion. One great dashboard > infinite mediocre ones. |
| Drag-and-drop everything | Looks cool, rarely used. Direct actions are faster. |
| Themes and skins | Distraction. Consistent experience > personalization. |
| Notification center | Badge anxiety. Show what matters on the dashboard. |
| Wizards for everything | Most tasks should be one screen, not five. |
| Advanced search syntax | "field:value" is for developers. Natural search for everyone. |
| Inline editing everywhere | Looks modern, causes accidents. Explicit edit mode is safer. |

### Complexity We Actively Avoid

- **No nested modals** (modal opens modal opens modal)
- **No more than 3 levels of navigation** (Today > Bookings > Booking #123)
- **No settings without defaults** (everything works out of the box)
- **No required fields that could have defaults** (reduce friction)

## Summary: The Tour CRM Design Manifesto

1. **Design for moments, not modules.** Organize by what users need to do, not how data is stored.

2. **Three clicks to anything.** If it takes more, redesign it.

3. **One primary action per screen.** Multiple equal options create paralysis.

4. **Recognition over recall.** Show options, don't make users remember.

5. **Speed is a feature.** Every millisecond of delay is a tiny frustration.

6. **Invisible complexity.** Power available, never forced.

7. **The interface should disappear.** Users should think about their work, not the software.

8. **Consistency is kindness.** Same patterns everywhere reduce learning.

9. **Feedback is mandatory.** Every action gets a response.

10. **When in doubt, leave it out.** Simplicity is the ultimate sophistication.
