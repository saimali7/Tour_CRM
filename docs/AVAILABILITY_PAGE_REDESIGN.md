# Availability Page Redesign

> Rethinking the Schedules page from first principles to serve its true purpose: **Capacity Planning & Booking Assistance**

---

## The Problem with "Schedules"

The current page is called "Schedules" and shows a list of tour instances. But this misses the point of why operators look at this page:

**They're not thinking:** "Show me schedule #47"
**They're thinking:** "Do we have space for 4 people on Saturday?"

---

## First Principles Analysis

### Who uses this page?

| User | Their Question | Current Answer |
|------|----------------|----------------|
| **Booking Agent (phone)** | "What's available Saturday afternoon?" | Must scroll through list, check each |
| **Manager** | "Are we over/under capacity this week?" | No aggregate view exists |
| **Owner** | "Which tours should we run more/less of?" | Must mentally calculate |

### What decisions need to be made?

1. **During a booking call:** "Which schedule should I recommend?"
2. **Planning:** "Do we need to add more Saturday tours?"
3. **Operations:** "Are we running empty tours we should cancel?"
4. **Revenue:** "Where should we push customers to maximize revenue?"

### The Mental Model Shift

```
OLD: "Schedules" = Administrative list of tour instances
NEW: "Availability" = Capacity inventory to sell
```

Think of it like:
- Hotel: Room availability calendar
- Airline: Seat inventory
- Restaurant: Table availability

---

## The New Design: "Availability"

### Rename
- **Sidebar:** Schedules â†’ **Availability**
- **URL:** /schedules â†’ /availability (with redirect)
- **Page Title:** "Availability & Capacity"

### Core Principle
**Start with the question, not the data.**

The page should answer: "What can I sell right now?"
